# API Specification — Public Interface Contract

## Purpose

This document defines the public API contract for Knowledge_Service. It specifies the endpoints, request/response formats, authentication model, versioning strategy, and error responses that applications use to interact with the platform.

## Scope

This document covers:
- API transport protocol and conventions
- Authentication and authorization model
- Endpoint specifications (conceptual — not implementation code)
- Request/response schemas
- Error response format
- Versioning strategy
- Rate limiting and throttling
- Pagination conventions

## Design Rationale

The API is designed around the principle that applications request knowledge, not operations. Endpoints are organized around knowledge concepts rather than provider actions. Applications never specify which providers to use; they specify what knowledge they need, and Knowledge_Service determines how to acquire it.

The API uses RESTful conventions over HTTP with JSON payloads for maximum compatibility across application platforms and languages.

## Transport Protocol

| Aspect | Specification |
|--------|--------------|
| Protocol | HTTPS (TLS 1.2+ required) |
| Base URL | Configurable per deployment (e.g., `https://ks.example.com/api/v1`) |
| Content Type | `application/json` for all requests and responses |
| Character Encoding | UTF-8 |
| Date Format | ISO 8601 (`YYYY-MM-DDTHH:mm:ssZ`) |

## API Versioning

### Strategy: URL Path Versioning

API versions are included in the URL path: `/api/v1/...`, `/api/v2/...`.

### Version Lifecycle

| Stage | Duration | Action Required |
|-------|----------|-----------------|
| Current | Indefinite | Active development and support |
| Deprecated | Minimum 2 major releases | Documentation updated; still functional |
| Sunset announced | 6 months minimum | Migration guide provided; new features disabled |
| Removed | After sunset period | Endpoint returns 410 Gone with migration URL |

### Version Compatibility Guarantees

- **Within a major version**: Backward compatible. New fields may be added to requests and responses. Existing fields are never removed or changed in meaning.
- **Across major versions**: No compatibility guarantee. Each major version is independently specified.

## Authentication Model

### API Key Authentication

Applications authenticate using API keys passed in the `Authorization` header:

```
Authorization: Bearer <api_key>
```

### API Key Lifecycle

| Stage | Description |
|-------|-------------|
| Created | Generated by administrator with scoped permissions |
| Active | Valid and authorized for API access |
| Rotated | Old key remains valid during transition period |
| Revoked | Immediately invalidated; all active sessions terminated |
| Expired | Inactive beyond configured lifetime; requires regeneration |

### Permission Scopes

API keys are scoped to specific capabilities:

| Scope | Description |
|-------|-------------|
| `knowledge:read` | Retrieve knowledge objects |
| `knowledge:search` | Search across stored knowledge |
| `knowledge:acquire` | Trigger knowledge acquisition requests |
| `sources:list` | View source registry entries |
| `health:read` | Access system health endpoints |

### Key Management

- API keys are managed by Knowledge_Service's administration interface (not exposed through the public API)
- Keys are hashed before storage; original values are never retrievable
- Key usage is logged for audit purposes
- Rate limits may be applied per key or per application identity

## Endpoints

### Knowledge Retrieval

**POST /api/v1/knowledge/retrieve**

Retrieve knowledge objects by identifier or query.

**Request:**
```json
{
  "ids": ["uuid-1", "uuid-2"],          // Optional: specific IDs to retrieve
  "query": "string",                     // Optional: search query (mutually exclusive with ids)
  "filters": {                           // Optional: filter criteria
    "source_ids": ["source-a", "source-b"],
    "min_confidence": 0.7,
    "date_range": {
      "from": "2026-01-01T00:00:00Z",
      "to": "2026-06-25T23:59:59Z"
    },
    "types": ["document", "chunk"]
  },
  "options": {                           // Optional: response options
    "include_markdown": true,
    "include_structured_data": false,
    "max_results": 50,
    "page_token": "string"               // For pagination
  }
}
```

**Response (200 OK):**
```json
{
  "results": [
    {
      "id": "uuid-1",
      "version": 1,
      "type": "document",
      "title": "...",
      "markdown": "...",
      "confidence": 0.92,
      "source_id": "nextjs-blog-001",
      "acquired_at": "2026-06-25T14:30:00Z",
      "citations": [...],
      ...
    }
  ],
  "pagination": {
    "has_more": false,
    "next_page_token": null,
    "total_count": 1
  },
  "metadata": {
    "request_id": "req-abc-123",
    "timestamp": "2026-06-25T14:31:00Z"
  }
}
```

**Error Responses:**
- `400 Bad Request`: Invalid request format or mutually exclusive fields provided
- `401 Unauthorized`: Missing or invalid API key
- `403 Forbidden`: API key lacks required scope
- `429 Too Many Requests`: Rate limit exceeded

### Knowledge Acquisition

**POST /api/v1/knowledge/acquire**

Request knowledge acquisition for a target. This is an asynchronous operation — the response acknowledges receipt, and results are available through retrieval.

**Request:**
```json
{
  "targets": [
    {
      "url": "https://example.com/article",
      "type": "web_page"
    }
  ],
  "options": {
    "freshness": "current",              // current, recent, any
    "min_confidence": 0.5,
    "max_wait_seconds": 300,
    "sources": ["preferred-source-1"],   // Optional: prefer specific sources
    "fallback_sources": ["fallback-1"]   // Optional: fallback if preferred fail
  }
}
```

**Response (202 Accepted):**
```json
{
  "acquisition_id": "acq-def-456",
  "status": "accepted",                  // accepted, processing, completed, failed
  "estimated_completion_seconds": 120,
  "metadata": {
    "request_id": "req-abc-123",
    "timestamp": "2026-06-25T14:31:00Z"
  }
}
```

Applications poll for results or use a separate status endpoint. Completed acquisitions are retrievable via the retrieve endpoint using the acquisition_id as a filter.

### Knowledge Search

**POST /api/v1/knowledge/search**

Search across stored knowledge using full-text and semantic search.

**Request:**
```json
{
  "query": "Next.js 15 new features",
  "filters": {
    "source_ids": ["tech-blogs"],
    "min_confidence": 0.7,
    "date_range": {
      "from": "2026-01-01T00:00:00Z"
    }
  },
  "options": {
    "limit": 20,
    "page_token": "string",
    "sort": "relevance",                 // relevance, date, confidence
    "include_snippets": true,            // Include context snippets around matches
    "semantic_weight": 0.6               // Weight for semantic vs full-text (0-1)
  }
}
```

**Response:** Same format as retrieve endpoint with paginated results.

### Source Registry Query

**GET /api/v1/sources**

List and query the source registry. Read-only access to source metadata.

**Query Parameters:**
- `page`: Page number (default: 1)
- `limit`: Results per page (default: 50, max: 200)
- `trust_min`: Minimum trust score filter (0.0 - 1.0)
- `topic`: Filter by topic expertise

**Response (200 OK):**
```json
{
  "sources": [
    {
      "id": "nextjs-blog-001",
      "name": "Next.js Blog",
      "url": "https://nextjs.org/blog",
      "type": "web_page",
      "trust_score": 0.95,
      "freshness_score": 0.88,
      "topics": ["web_frameworks", "javascript"],
      "status": "healthy"
    }
  ],
  "pagination": { ... },
  "metadata": { ... }
}
```

## Error Response Format

All error responses follow a consistent format:

```json
{
  "error": {
    "code": "INVALID_REQUEST",
    "message": "Field 'query' is required when 'ids' is not provided.",
    "request_id": "req-abc-123",
    "details": {
      "field": "query",
      "issue": "missing_required_field"
    }
  },
  "metadata": {
    "timestamp": "2026-06-25T14:31:00Z"
  }
}
```

### Error Codes

| Code | HTTP Status | Meaning |
|------|------------|---------|
| `INVALID_REQUEST` | 400 | Request format or validation error |
| `UNAUTHORIZED` | 401 | Authentication failure |
| `FORBIDDEN` | 403 | Authorization failure (insufficient scope) |
| `NOT_FOUND` | 404 | Requested resource does not exist |
| `RATE_LIMITED` | 429 | Too many requests |
| `ACQUISITION_FAILED` | 502 | All acquisition sources failed |
| `INTERNAL_ERROR` | 500 | Unexpected system error |
| `SERVICE_UNAVAILABLE` | 503 | System temporarily unavailable |
| `GONE` | 410 | API version has been sunset |

### Error Response Fields

| Field | Type | Description |
|-------|------|-------------|
| `code` | String | Machine-readable error code |
| `message` | String | Human-readable error description |
| `request_id` | String | Trace ID for debugging and support |
| `details` | Object | Optional: additional context about the error (field-level validation errors, etc.) |

## Rate Limiting

### Strategy

Rate limiting is applied per API key with configurable limits. Limits are defined in configuration, not hardcoded.

### Headers

Every response includes rate limit information:

```
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1719321600
Retry-After: 30          (only when rate limited)
```

### Limit Tiers

| Tier | Requests/Minute | Requests/Hour | Burst |
|------|-----------------|---------------|-------|
| Default | Configurable | Configurable | Configurable |
| Elevated | Higher limits | Higher limits | Higher burst |
| Internal | Unlimited (service-to-service) | N/A | N/A |

Tier assignment is based on API key scope and application identity.

## Pagination

### Cursor-Based Pagination

All list endpoints use cursor-based pagination for consistency and performance:

**Request:** Include `page_token` from previous response's `pagination.next_page_token`.

**Response:**
```json
{
  "pagination": {
    "has_more": true,
    "next_page_token": "eyJwYWdlIjoyLCJzb3J0IjoiMTY4NzY1NDMyMSJ9",
    "total_count": 1500
  }
}
```

### Token Format

Page tokens are opaque base64-encoded strings. Applications must not parse or modify them. Tokens encode the cursor position and sort order used for the current page.

## Request ID and Tracing

Every request receives a unique `request_id` that is:
1. Generated by the API Layer on receipt
2. Included in all response metadata (success and error)
3. Propagated through all internal layers
4. Used to correlate logs, metrics, and traces across the system

Applications should include this ID in support requests for debugging.

## Field Omission Policy

To reduce payload size and API surface:
- Fields with null/empty values are omitted from responses (not included as `null`)
- Optional fields are only included when they contain meaningful data
- The `include_*` options in request objects allow applications to control which optional fields are returned

## Assumptions

- JSON is the primary payload format; other formats may be added later via content negotiation
- All timestamps are UTC
- UUIDs follow RFC 4122 v4 (or v7 for sortability)
- Maximum request body size: 1 MB
- Maximum response body size: 10 MB (larger results require pagination)

## Future Evolution

Future phases may add:
- GraphQL endpoint for flexible field selection
- WebSocket/SSE endpoints for streaming acquisition progress
- Batch operations (acquire multiple targets in one request)
- Webhook notifications for async acquisition completion
- Export endpoints (download knowledge objects in various formats)
- Admin endpoints for source registry management

All future endpoints follow the same authentication, error handling, and versioning conventions defined in this specification.
